# ⚠️ 성능 이슈 분석 및 개선 제안 보고서

현재 프로젝트의 프론트엔드(화면 로딩) 및 백엔드(API 응답 속도) 성능 저하 원인을 분석한 결과, 다음과 같은 심각한 병목 현상들이 발견되었습니다. PM(기획/관리자) 및 개발 조직의 빠른 인지를 위해 공유합니다.

---

## 🔍 1. 백엔드 서비스 응답 속도 저하 원인 (API 렌더링 지연)

### 🚨 [원인 1] GET 요청(조회) 시 발생하는 N+1 업데이트 쿼리 (가장 심각함)
- **위치:** `src/app/api/singers/[id]/route.ts` (GET 메서드)
- **현상 및 문제점:** 가수의 전체 공연 정보를 가져온 뒤 과거 공연 상태를 업데이트하기 위해 `Promise.all` 내에서 개별적으로 `update` 쿼리를 호출하고 있습니다.
- **결과:** 조회를 위한 GET 요청에서 수십 번의 **DB 쓰기(UPDATE) 작업이 동시다발적으로 발생**하여 백엔드 응답 시간을 극단적으로 지연시킵니다.

### 🚨 [원인 2] 데이터 페이로드 비대화 및 페이징 부재
- **위치:** `src/app/api/singers/[id]/route.ts`
- **현상 및 문제점:** 가수의 모든 과거/현재 공연 내역과 노래 목록을 **조건이나 개수 제한 없이 한 번에 모두 로딩**합니다. 시간이 지날수록 전송해야 하는 데이터 크기가 커져 네트워크 병목을 유발합니다.

---

## 🔍 2. 프론트엔드 화면 로딩 지연 원인

### 🚨 [원인 3] 비효율적인 중복 API 호출
- **위치:** `src/hooks/useSingerResources.ts`
- **현상 및 문제점:** `useSongs`와 `usePerformances` 커스텀 훅이 동일한 API(`/api/singers/${singerId}`)를 독립적으로 각각 호출합니다. 한 화면에서 이 두 훅을 동시에 사용하면 **동일하고 무거운 API가 2번 중복해서 호출**되어 서버 부하가 2배로 증가하고 화면 렌더링을 늦춥니다.

### 🚨 [원인 4] 최적화되지 않은 원본 이미지 강제 렌더링
- **위치:** `src/app/explore/page.tsx` 및 `singer/dashboard/page.tsx` 등 다수
- **현상 및 문제점:** 썸네일이나 프로필 이미지를 렌더링할 때 HTML 표준 `<img src="...">` 태그를 사용하여, 대용량 원본 이미지를 그대로 브라우저가 다운받게 만듭니다. 이로 인해 초기 화면 렌더링(LCP)이 크게 저하됩니다.

## 💡 3. 개발팀 필수 권고 조치 사항 (Action Plan)

1. **상태 업데이트 로직 백그라운드 분리 (Backend)**
   - API 조회(GET) 시 발생하던 상태 업데이트 로직을 제거하고, Vercel Cron 등의 별도 스케줄러를 도입하여 1분/5분 주기로 백그라운드에서 한 번에 DB 상태를 업데이트하도록 구조를 변경해야 합니다.
2. **API 페이징 및 데이터 선별 조회 (Backend)**
   - 무거운 데이터 모델을 반환할 때 페이지네이션을 적용하거나, "최근 N개" 또는 "진행중인 공연" 단위로 필터링하여 페이로드를 가볍게 유지합니다.
3. **프론트엔드 상태/요청 캐싱 (Frontend)**
   - React Query나 SWR을 도입하여 `useSingerResources.ts` 내의 API 중복 호출을 막고 1회의 요청으로 노래와 공연 데이터를 받아 캐싱 처리해야 합니다.
4. **Next.js `<Image>` 컴포넌트로 전면 교체 (Frontend)**
   - 프로젝트 내 모든 `<img>` 태그를 `next/image` 로 교체하여 자동 리사이징, WebP 포맷 변환, Lazy Loading 적용을 통해 렌더링 블로킹을 해소해야 합니다.
