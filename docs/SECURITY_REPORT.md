# Busking 플랫폼 보안 취약점 및 개선 방안 보고서

본 문서는 현재 Busking 플랫폼의 주요 코드베이스를 점검하여 발견된 심각한 보안 취약점과 이에 대한 해결 방안을 PM 및 개발팀이 인지하고 신속히 대응할 수 있도록 작성되었습니다.

## 1. API 라우트 인증/인가(Authorization) 누락 [Critical]
현재 대부분의 `/api/*` 경로(POST, DELETE 등)에서 요청을 보낸 사용자가 누구인지 검증하는 과정이 누락되어 있습니다.
- **위험성:** 악의적인 사용자가 다른 가수의 ID로 임의의 곡을 추가/삭제하거나, 허위 공연을 등록하여 타인의 포인트를 무단으로 차감시킬 수 있습니다.
- **해결 방안:** 모든 상태 변경 API 핸들러 최상단에 사용자 로그인 여부 및 권한을 철저히 검증하는 로직(`auth()` 기반)을 추가해야 합니다.

## 2. 실시간 채팅 서버 (Realtime-Server) 검증 부재 [High]
관객의 편의성을 위해 익명 소통을 허용하는 기획 의도는 유지하되, 현재 발생할 수 있는 보안 위협에 대한 최소한의 안전장치(Compensating Controls)가 필요합니다.
- **위험성:** 현재 누구나 소켓 서버에 접속하여 가수 행세를 하거나, 공지문 띄우기, 허위 후원 띄우기 등의 관리자 시스템 이벤트를 조작할 수 있습니다.
- **해결/대체 방안:**
  - **가수와 관객의 엄격한 소켓 이벤트 권한 분리:** 관객은 오직 채팅(`send_message`)과 신청곡(`song_requested`)만 가능하도록 서버단에서 강제 제한.
  - **게스트용 1회성 토큰 발급:** 봇/외부 툴을 이용한 무단 소켓 접속을 막기 위해, 정상적인 프론트엔드 웹을 거친 관객에게만 짧은 만료 시간의 서명된 게스트 토큰 발급.
  - **CORS 오리진 락(Target Restriction):** 지정된 웹사이트 도메인에서만 소켓 연결을 수락하도록 설정 변경.
  - **DDoS/도배 방지:** 초당 메시지 전송 제한(Rate-Limit) 적용.

## 3. Next.js Middleware의 라우트 보호 미흡 [Medium]
미들웨어 단에서 비공개 페이지들에 대한 일괄적인 보호 로직이 작동하지 않고 통과 처리되고 있습니다.
- **위험성:** 개별 페이지 및 API단에서 검증을 깜빡할 경우 누구나 비공개 페이지에 직접 접속할 수 있는 취약점이 발생합니다.
- **해결 방안:** `isPublicRoute`가 아닌 경우 `auth().protect()`를 호출하도록 미들웨어 로직 보완이 필요합니다.

## 4. 악의적 입력값 기반 공격 (XSS/SQLi) 모니터링 [Medium]
- **현황:** 프레임워크(Next.js, Prisma) 레벨에서 1차적인 방어가 이루어지고 있으나, 일부 테스트 코드에서 `$queryRawUnsafe`가 사용되고 있습니다.
- **해결 방안:** 운영 환경(Production)에서 안전하지 않은 쿼리가 실행되지 않도록 테스트 코드 및 소켓 기반 입력(채팅 메시지 렌더링) 부분에 대한 XSS 방어(Escape)를 지속 점검해야 합니다.

**최종 권고사항:** 본 리포트의 1, 3, 4번 이슈는 필수적으로 즉각 보완해야 하며, 2번 이슈의 경우 기획 파트와 조율하여 제안된 접근법(CORS, 게스트 권한 분리 등)을 적용하는 것을 강력히 권장합니다.
