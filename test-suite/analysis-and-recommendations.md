# 📋 Analysis & Future Recommendations

After reviewing the current implementation of the live performance system, here are the key findings and suggested improvements for the next phase of development.

## 1. Time & Status Synchronization 🕒
- **Current Issue**: The `status` field in the database ('scheduled', 'live', 'completed') is the primary driver, but the UI also relies heavily on `new Date()` calculations.
- **Problem**: If a singer starts a performance early or late, the "Upcoming" or "Live" lists might show inconsistent data if they rely purely on the `status` field without checking the current time (and vice versa).
- **Recommendation**: 
  - Centralize the "isLive" logic into a single utility function used by both the Backend and Frontend.
  - Implement a **Cron Job** (e.g., via Vercel Cron or a simple `setInterval` on the server) that sweeps the database every minute to update `scheduled` -> `live` and `live` -> `completed` based on `endTime`.

## 2. Chat Server Resilience 💬
- **Current Issue**: The chat server is a standalone Node.js process. If it crashes, the Next.js app doesn't have a fallback, and users might see a "Connecting..." state forever.
- **Recommendation**:
  - Add a "Chat Status" indicator in the Singer Dashboard.
  - Implement a fallback "System Notification" mode that uses polling if WebSockets are unavailable.

## 3. Audience Disconnection Flow 🚪
- **Current Issue**: When a performance ends, the singer is redirected, but the audience might stay on the page until they refresh.
- **Recommendation**:
  - The `performance_ended` socket event should trigger a "Thank You" modal for the audience and a redirect to the singer's profile after 10 seconds.
  - Automatically clear the `viewingCount` in Redis when the performance status changes to `completed`.

## 4. Timezone Safety 🌍
- **Current Issue**: `new Date()` on the server might be UTC, while the user's browser is local time.
- **Recommendation**: Always store `startTime` and `endTime` in **ISO 8601 (UTC)**. Ensure the frontend converts these to the local user's timezone only at the point of display.

## 5. Security & Validation 🔒
- **Current Issue**: The `api/song-requests` endpoint allows anyone to send a request if they know the `performanceId`.
- **Recommendation**:
  - Verify that the `performanceId` is currently `live` before accepting a request.
  - Add rate-limiting to prevent "spam" requests from a single user.
